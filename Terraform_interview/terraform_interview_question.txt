1. tfsec
tfsec is a static security scanner specifically designed for Terraform code.
It analyzes .tf files and detects security misconfigurations before infrastructure is deployed.

****Example Problems tfsec Detects

   Public S3 buckets
   Open security groups (0.0.0.0/0)
   Unencrypted storage
   Public databases
   Missing logging

2. Checkov
Checkov is a more advanced Infrastructure as Code security scanner.
***It supports multiple IaC frameworks:
   Terraform
   Kubernetes
   CloudFormation
   Docker
   Helm
**Example
  Run scan:
   checkov -d .
Checkov checks against hundreds of security policies.
**Example checks:
   Encryption enabled
   Logging enabled
   No public storage
   IAM least privilege
   Secure networking

3. Open Policy Agent (OPA)
Open Policy Agent is a policy engine used to enforce custom rules across infrastructure, Kubernetes, APIs, and CI/CD.
Unlike tfsec and Checkov (which scan), OPA enforces policies before deployment.
OPA uses Rego language.
***Where OPA is Used
Kubernetes admission controller
CI/CD pipelines
API authorization
Terraform policy validation

4. Sentinel
HashiCorp Sentinel is a policy-as-code framework built by HashiCorp used mainly with Terraform Enterprise / Terraform Cloud.
It enforces organizational security rules before Terraform applies infrastructure.
Where Sentinel Runs
Inside:
Terraform Cloud
Terraform Enterprise

Q3 .how can modulizse terraform function, variable
Answer:
In Terraform we modularize infrastructure by creating reusable modules that 
contain main.tf, variables.tf, and outputs.tf. Variables are used to parameterize 
the module so it can be reused across multiple environments like dev, staging, 
and production. The root module calls these child modules and passes values 
through variables or tfvars files, making the infrastructure code reusable, 
maintainable, and scalable.

Q4 . where to use count , for_each and in terraform
Answer
In Terraform, count is used to create multiple identical resources using an index value, 
while for_each is used when each resource needs unique configurations based on a 
list or map. In production environments we usually prefer for_each because 
it provides better resource management and avoids index shifting issues.

| Feature            | `count`             | `for_each`               |
| ------------------ | ------------------- | ------------------------ |
| Input              | Number              | List / Map               |
| Resource reference | index               | key                      |
| Flexibility        | Less                | More                     |
| Use case           | identical resources | different configurations |

Important Terraform Rule
You cannot use both count and for_each together in the same resource.


question - how  to use for_each with map in terraform
ans- To use for_each with a map in Terraform, you can define a resource that iterates over the map's key-value pairs.
For example, if you have a map of server names and their corresponding IP addresses, you can create resources for each server like this:
variable "servers" {
  type = map(string)
  default = {
    server1 = "lks"  
      server2 = "lks"   
   }     
}

resource "aws_instance" "example" {
  for_each = var.servers

  ami           = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
    IP   = each.value
  }
}
In this example, the aws_instance resource will be created for each entry in the servers map, with the instance name set to the key and the IP address set to the value from the map.

"How do you manage Terraform state for multiple environments using the same module?"

Answer:

We use reusable Terraform modules and separate environment folders like dev, stage, and prod. Each environment calls the same module
 but uses a different backend key in Azure Storage so each environment maintains an isolated Terraform state file.

question - how to identify the drift in terraform
ans - to identify drift in Terraform, you can use the `terraform plan` command. 
This command compares the current state of your infrastructure with the desired state
 defined in your Terraform configuration files. I
   if there are any differences (drift) between the actual infrastructure and the configuration,
    Terraform will show those changes in the output of the plan. You can also use `terraform show`
     to inspect the current state and compare it with your configuration to identify any discrepancies.
and you can also use refresh command to update the state file with the current infrastructure state before running plan to get accurate drift detection.
