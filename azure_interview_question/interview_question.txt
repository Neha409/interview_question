question - what is difference b/w l7 load balancer and l4 load balancer in azure or application gateway and load balancer in azure
ans - L7 Load Balancer (Application Gateway) operates at the application layer (Layer 7) of the OSI model. 
It can make routing decisions based on the content of the HTTP/HTTPS requests, such as URL paths, host headers,
 and cookies. It is ideal for web applications and can provide features like SSL termination, session affinity, 
 and Web Application Firewall (WAF) capabilities. 
L4 Load Balancer (Azure Load Balancer) operates at the transport layer (Layer 4) of the OSI model.
 It routes traffic based on IP address and port without inspecting the content of the traffic. 
 It is suitable for non-HTTP workloads, such as TCP and UDP applications, and provides high performance and low latency.

question - what is azure function and how it works
ans - Azure Functions is a serverless compute service that allows you to run event-driven code without
    having to manage infrastructure. It works by executing small pieces of code, called "functions," in response 
    to various triggers, such as HTTP requests, timers, or messages from other Azure services. 
    When a trigger event occurs, Azure Functions automatically allocates the necessary resources to 
    execute the function and scales it based on demand.
    for example, you can create a function that processes data from an Azure Storage Blob whenever 
    a new blob is added, or you can create an HTTP-triggered function that responds to web requests.

question- what is differnenc b/w azure function and logic app in azure
ans- Azure Functions is a serverless compute service that allows you to run code in response to
events, while Azure Logic Apps is a cloud service that helps you automate workflows and integrate applications, data, and services.
Azure Functions is more focused on executing code and is ideal for scenarios where you need to run custom
logic in response to events, such as processing data or handling HTTP requests.
Azure Logic Apps, on the other hand, is designed for building complex workflows and integrating various services

without writing code. It provides a visual designer and a wide range of connectors to easily create workflows that can automate tasks, 
integrate with other services, and orchestrate business processes. 
In summary, Azure Functions is best for running code in response to events, while Azure Logic Apps is best for automating workflows 
and integrating services without writing code.


Question - difffernce b/w NSG and ASG in azure
ans- NSG (Network Security Group) is a security feature in Azure that allows you to
control inbound and outbound traffic to and from Azure resources. It contains a list of security rules that allow or deny 
traffic based on source and destination IP addresses, ports, and protocols. NSGs are typically associated with subnets or 
 individual network interfaces.
ASG (Application Security Group) is a feature in Azure that allows you to group virtual machines and define security rules
 based on those groups. ASGs provide a way to manage security rules for multiple virtual machines that share 
 similar security requirements. You can create ASGs and then associate them with NSGs to apply security rules to 
 the grouped virtual machines. ASGs are particularly useful for managing security in large environments 
 where you have many virtual machines with similar security needs, as it allows you to simplify the management  
 of security rules by grouping resources together.

question- what is vnet peering in azure
    ans- VNet peering in Azure is a networking feature that allows you to connect two virtual networks (VNets) in the same Azure region. 
    This connection enables resources in the peered VNets to communicate with each other as if they were part of the same network. 
    VNet peering provides low-latency, high-bandwidth connectivity between the peered VNets, allowing for seamless communication and data transfer. 
    It is commonly used to enable communication between different applications or services that are deployed in separate VNets, while still maintaining isolation and security between them. 
    VNet peering can be established using either the Azure portal, Azure PowerShell, or Azure CLI, and it does not require any additional gateways or VPNs to set up.
    3 types of vnet peering in azure
1. Global VNet Peering: This allows you to peer virtual networks across different Azure regions
2. VNet Peering within the same region: This allows you to peer virtual networks within the same Azure region
3. VNet Peering with Azure Virtual WAN: This allows you to peer virtual networks with Azure Virtual WAN, which is a networking service that provides optimized and automated branch connectivity to Azure.

question- what service princple in azure
    ans- A Service Principal in Azure is an identity created for use with applications, hosted services, and 
    automated tools to access Azure resources. 
    It is a security identity that allows applications to authenticate and access resources in Azure without 
    the need for user credentials.
    Service Principals are commonly used in scenarios such as automation, 
    continuous integration/continuous deployment (CI/CD) pipelines, and when applications need to access Azure resources 
    on behalf of a user or another application. 
    They can be assigned specific roles and permissions to control access to resources, ensuring that applications have 
    the necessary permissions while adhering to the principle of least privilege. and user need to rotate the credentials of 
    service principal manually.

Question how to can migrated on-premises resources to auzre 
    ans- To migrate on-premises resources to Azure, you can use the Azure Migrate service, which provides a centralized hub for discovering,
     assessing, and migrating on-premises applications and workloads to Azure. 
    The migration process typically involves the following steps:   
1. Discovery: Use Azure Migrate to discover and inventory your on-premises resources, including servers, applications, and databases.
2. Assessment: Assess the readiness of your on-premises resources for migration to Azure, including
    performance, compatibility, and cost analysis.
3. Migration: Use Azure Migrate to migrate your on-premises resources to Azure, which
    can be done using various migration tools and services, such as Azure Site Recovery for virtual machines, 
    Azure Database Migration Service for databases, and Azure App Service Migration Assistant for web applications.
