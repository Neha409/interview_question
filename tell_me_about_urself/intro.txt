question-How do you troubleshoot a pod in CrashLoopBackOff state in Kubernetes (AKS)?

To troubleshoot a pod in CrashLoopBackOff state in Kubernetes (AKS), I would follow these steps:
“When a pod is in CrashLoopBackOff state, it means the container is repeatedly failing and Kubernetes is restarting it with backoff delay.
My first step is to check the pod status and restart count using kubectl get pods. Then I use kubectl describe pod to review events 
and identify any errors like OOMKilled, image pull issues, or probe failures.
Next, I check the container logs using kubectl logs to understand the exact reason for the crash. If the container is restarting too 
quickly, I also check previous logs using --previous.
I verify common causes such as resource limits (CPU/memory), configuration issues (ConfigMaps/Secrets), environment variables, 
or missing dependencies. For example, if it’s OOMKilled, I increase memory limits accordingly.
I also check liveness and readiness probes, as incorrect probe configuration can cause repeated restarts.
Once I identify the root cause, I fix the issue, test in a lower environment if needed, and then redeploy. Finally, 
I monitor the pod to ensure stability.”


Next Question (Terraform – Very Important 🔥)

👉 How do you handle Terraform state and avoid conflicts when multiple people are working on the same infrastructure?


“Terraform state is critical because it keeps track of the current infrastructure. To handle state safely and avoid conflicts when
 multiple users are working, I use a remote backend in Azure Storage Account with Blob containers.
This setup provides state locking, which ensures that only one person can modify the state at a time, preventing concurrent changes
 and corruption.
I also enforce RBAC policies to control access to the storage account, ensuring only authorized users or pipelines can modify the state.
Additionally, I organize state files based on environments like dev, staging, and production, either using separate backend configurations
or workspaces, to avoid accidental changes across environments.
We also enable versioning and soft delete on the storage account, so we can recover previous state versions if needed.
Overall, this approach ensures safe collaboration, prevents conflicts, and maintains consistency in infrastructure provisioning.”


🎤 Next Question (CI/CD + Security – Very Important 🔥)

👉 How do you secure secrets in CI/CD pipelines?

In Jenkins, I use the credentials store to securely manage sensitive data like passwords, tokens, and keys, and reference 
them in pipelines without hardcoding.
In GitHub Actions, I use the Secrets section to store sensitive values, which are encrypted and automatically masked in logs.
For enterprise-grade security, I integrate external secret management tools like Azure Key Vault or HashiCorp Vault. Instead 
of storing secrets directly in pipelines, I fetch them dynamically at runtime using secure authentication methods like Managed Identity or service principals.
I also ensure role-based access control (RBAC) so only authorized pipelines or users can access specific secrets. Additionally, 
I follow best practices like regular secret rotation, avoiding exposure in logs, and never storing secrets in code repositories.
This approach ensures that secrets are securely managed, centrally controlled, and compliant with security standards.”
